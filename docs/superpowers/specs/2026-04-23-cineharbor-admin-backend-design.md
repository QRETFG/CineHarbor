# CineHarbor 后端与管理后台设计

**日期：** 2026-04-23  
**目标：** 在现有 CineHarbor React 前端项目中新增 `Express + SQLite` 后端和 `/admin` 管理界面，实现管理员登录、影视数据管理、素材文件上传，以及前端零散静态素材上传。

## 1. 需求边界

### 已确认范围
- 保留现有 `CineHarbor` React 前端项目作为前台站点
- 新增同仓全栈架构：前端继续使用 React，后端新增 `Node.js + Express + SQLite`
- 新增专门的后台管理界面 `/admin`
- 后台需要账号密码登录
- 后台需要支持：
  - 影视条目数据管理
  - 图片素材上传
  - 前端零散静态素材上传
- 不需要上传整套前端构建产物，不做整站 `dist.zip` 发布替换
- 前台文案和布局仍保留在 React 代码中，不做在线 CMS 化

### 明确排除项
- 不支持通过后台上传整站前端代码或构建包
- 不支持在线编辑 React 页面结构或组件逻辑
- 不引入外部数据库服务，数据库固定为 SQLite
- 不做多角色权限系统，初版仅支持管理员账号

## 2. 当前项目状态

- 当前 `CineHarbor` 是一个纯前端 Vite + React + TypeScript 项目
- 影视数据当前写死在 `src/data/movies.ts`
- 前端页面已具备影视列表展示能力，但没有后端 API、登录、数据库和后台管理界面
- 项目已有本地素材目录 `public/images/...`，适合迁移到“数据库记录 + 文件存磁盘”的模式

## 3. 方案对比与选型

### 方案 A：单 React 应用 + Express + SQLite（采用）
- 现有 React 应用中增加 `/admin` 路由
- 同仓新增 `Express` 服务提供 API 和上传能力
- 优点：
  - 与现有技术栈最一致
  - 开发和部署复杂度最低
  - 适合当前项目体量
- 风险：
  - 后台规模继续增长时，前后台边界需要通过目录结构保持清晰

### 方案 B：同仓双前端 + 独立后台前端
- 优点：前后台界面边界更清晰
- 缺点：需要新增单独管理端构建链路，对当前项目属于过度设计

### 方案 C：Express API + 服务端模板后台
- 优点：后台初期实现轻
- 缺点：与现有 React 项目割裂，长期维护体验较差

最终采用 **方案 A**。

## 4. 总体架构

系统拆分为三层：

1. **前台站点**
   - 继续使用现有 React 页面
   - 影视数据改为通过 API 获取
   - 上传后的素材通过公开 URL 被前台引用

2. **后台管理界面**
   - 挂载在 `/admin`
   - 提供登录页、仪表盘、影视管理页、素材库页、前端素材上传页
   - 与后端 API 通过同域请求交互

3. **后端服务**
   - 使用 `Express`
   - 提供登录鉴权、影视 CRUD、素材上传、后台统计接口
   - 负责初始化 SQLite 数据库和上传目录
   - 对上传文件提供静态访问路径

## 5. 数据设计

### 5.1 `admins`
用于后台管理员账号。

字段建议：
- `id`
- `username`
- `password_hash`
- `created_at`
- `updated_at`
- `last_login_at`

约束：
- `username` 唯一
- 密码仅存 `bcrypt` 哈希

### 5.2 `assets`
用于统一管理上传素材元数据，文件本体不存入 SQLite，仅落本地磁盘。

字段建议：
- `id`
- `kind`：`poster` / `backdrop` / `site_asset`
- `original_name`
- `stored_name`
- `mime_type`
- `size_bytes`
- `storage_path`
- `public_url`
- `uploaded_by`
- `created_at`

### 5.3 `movies`
用于影视主数据。

字段建议：
- `id`
- `title`
- `year`
- `rating`
- `genre`
- `description`
- `external_url`
- `poster_asset_id`
- `backdrop_asset_id`
- `status`：`draft` / `published`
- `created_at`
- `updated_at`

### 5.4 `movie_sections`
用于控制首页区块归属和排序，避免同一影视记录被重复存储。

字段建议：
- `id`
- `movie_id`
- `section_key`：`featured` / `popular` / `recommended`
- `sort_order`

### 5.5 数据关系
- 一个 `movie` 可关联一个海报素材和一个背景素材
- 一个素材文件可被多个影视记录复用
- 一个影视记录可出现在多个首页区块

## 6. 存储设计

### SQLite
- 数据库存储于项目本地，例如 `storage/data/cineharbor.sqlite`
- 适合单机部署、轻量后台和低运维复杂度场景

### 文件存储
- 上传文件存放在本地磁盘，例如：
  - `storage/uploads/posters`
  - `storage/uploads/backdrops`
  - `storage/uploads/site-assets`
- 数据库中仅保存元数据和公开访问路径

### 设计原则
- 结构化数据进 SQLite
- 二进制文件进磁盘
- 前端通过 URL 访问素材，避免把文件内容塞进数据库

## 8. 鉴权设计

### 登录方案
- 管理员使用账号密码登录
- 后端校验 `bcrypt` 哈希密码
- 登录成功后写入 `httpOnly` 会话 Cookie

### 会话策略
- 后台接口统一挂载鉴权中间件
- 未登录访问 `/api/admin/*` 时返回 `401`
- 前端收到未授权响应后跳转回登录页

### 初版安全边界
- 初版仅支持单管理员体系
- 不做细粒度角色权限划分
- 不将认证 token 存储在浏览器 `localStorage`

## 9. API 设计

### 9.1 鉴权接口
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 9.2 后台统计接口
- `GET /api/admin/dashboard`

返回内容包括：
- 影视数量
- 素材数量
- 最近上传列表
- 最近更新影视列表

### 9.3 影视管理接口
- `GET /api/movies`
- `GET /api/movies/:id`
- `POST /api/movies`
- `PUT /api/movies/:id`
- `DELETE /api/movies/:id`

### 9.4 素材接口
- `GET /api/assets`
- `POST /api/assets`
- `DELETE /api/assets/:id`

### 9.5 前台读取接口
- 前台首页可复用 `GET /api/movies`
- 后续可按区块或状态增加筛选参数

## 10. 上传流程设计

### 流程 A：影视相关素材上传
1. 管理员进入素材库上传海报或背景图
2. 后端校验文件类型、大小、扩展名
3. 文件落到对应上传目录
4. 数据库写入 `assets` 元数据
5. 管理员在影视编辑表单中选择素材进行关联

### 流程 B：前端零散素材上传
1. 管理员进入“前端素材上传”页
2. 上传图片或静态文件
3. 后端保存文件并返回公开访问路径
4. 前台代码后续引用该路径使用资源

### 上传约束
- 使用 `multipart/form-data`
- 按素材类型写入不同目录
- 校验 MIME、扩展名和文件大小
- 不允许任意文件直接写入站点根目录

## 11. 错误处理设计

### 后台交互错误
- 登录失效：跳回登录页并提示重新登录
- 表单校验失败：在字段级展示错误信息
- 文件上传失败：保留当前表单内容，允许重试

### 资源一致性
- 删除素材前检查是否被影视记录引用
- 被引用素材不可直接删除，必须先解绑

### 系统错误
- 数据库写入失败时返回明确错误信息
- 文件写入失败时记录日志并返回失败原因
- 前台和后台都不吞错，保留可追踪的错误提示

## 12. 初始化与迁移设计

### 启动初始化
- 后端启动时自动创建 SQLite 文件
- 自动创建缺失的数据表
- 自动创建上传目录

### 首个管理员初始化
- 通过环境变量或单独脚本初始化管理员账号
- 避免把明文密码写死进源码

### 现有数据迁移
- 提供一次性导入脚本
- 将 `src/data/movies.ts` 中的静态数据迁移到 SQLite
- 前台改为优先读取 API 数据

### 后端验证
- 登录成功与失败场景
- 未登录访问后台接口被拦截
- 影视 CRUD 正常
- 素材上传成功
- 被引用素材删除被拒绝

### 前端管理端验证
- 登录流程可用
- 影视新增、编辑、删除可用
- 上传素材后列表刷新并可被影视表单引用

### 人工验收流程
1. 登录后台

3. 新增影视条目并关联海报

## 15. 风险与控制

- **SQLite 并发能力有限**：当前项目是轻量后台，单机使用可接受
- **文件目录被误删**：启动时自动补建目录，并通过数据库元数据减少孤儿文件
- **旧静态数据与新数据库并存**：通过迁移脚本和 API 优先读取策略平滑切换
- **后台功能膨胀**：初版仅做数据与素材管理，不把项目推向 CMS 化

## 16. 完成标准

当以下条件满足时，视为该设计达成目标：

1. 项目具备可运行的 `Express + SQLite` 后端
2. 管理员可登录 `/admin` 后台
3. 后台可管理影视数据
4. 后台可上传和管理海报、背景图、站点素材
5. 前台可从 API 读取影视数据
6. 零散前端素材可通过后台上传并返回可引用路径
